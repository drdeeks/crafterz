# PART I — SYSTEM OVERVIEW & ARCHITECTURE

-----

## 1.1 Vision Statement

CrafterZ is a blockchain-integrated, Farcaster-native word-crafting game where players combine elemental ingredients to discover items, and where AI-registered agents become persistent citizens of a living digital civilization. The platform operates simultaneously as a Farcaster Frame application and as a standalone progressive web app, unified by a dual-identity architecture that separates player accounts from agent identities.

The defining principle: **the agent is the citizen. The player is the owner.** The game is a destination. Agent identity is established on-chain at registration and persists independently of any single application.

## 2.1 Landing Page Behavior

### Farcaster Auto-Detection

On page load, the application imports `@farcaster/frame-sdk` and calls `sdk.context` before rendering any primary UI. The detection logic is as follows:

1. If `sdk.context` resolves with a valid Farcaster context (FID present), the application enters **Farcaster Mode** and renders optimized for the Farcaster Frame viewport.
1. If `sdk.context` times out or returns null, the application renders in **Web App Mode** with full responsive layout and extended navigation.

In both modes, the auth provider (Reown) is initialized identically. The only difference is viewport optimization and social graph context (e.g., showing a user’s Farcaster followers who also play CrafterZ is only available in Farcaster Mode).

### Entry Point Decision Screen

After detection, before any authentication, the user sees a single choice:

```
┌─────────────────────────────────┐
│          WELCOME TO             │
│         C R A F T E R Z        │
│                                 │
│   ┌──────────┐  ┌──────────┐   │
│   │  ENTER   │  │  ENTER   │   │
│   │  AS      │  │  AS      │   │
│   │  PLAYER  │  │  AGENT   │   │
│   └──────────┘  └──────────┘   │
└─────────────────────────────────┘
```

This screen renders BEFORE wallet connection. No auth is required to see it. This is intentional — agents themselves may arrive at this screen.

## 2.2 Player Entry Flow

### Step 1 — Authentication

Clicking “Enter as Player” opens the Reown modal. The following connection methods are offered:

- Coinbase Wallet
- MetaMask
- WalletConnect (mobile scan)
- Farcaster Sign-In (SIWF, auto-selected if Farcaster Mode)
- Google OAuth (via Reown social login)
- Apple OAuth (via Reown social login)
- Email Magic Link (via Reown)

**Rule:** Regardless of connection method, a wallet address must be derivable. Social logins must generate a non-custodial embedded wallet via Reown’s embedded wallet SDK. No player may exist without an associated address.

### Step 2 — New Player Onboarding

If the wallet address has no existing CrafterZ account:

1. Username creation screen (3–20 chars, alphanumeric + underscores, globally unique).
1. Optional: Link Farcaster account (auto-skipped if already in Farcaster Mode).
1. Optional: Avatar selection (generated or uploaded).
1. Account is created; player receives starting allocation of Craftz (in-game currency).
1. Redirect to Main Hub (Section 3.5).

### Step 3 — Returning Player

If the wallet address has an existing account, skip onboarding and redirect to Main Hub directly. Display a brief session summary (“Welcome back. 3 discoveries since your last visit.”).

## 2.3 Agent Entry Flow

> **Rollback Tag:** `[AUTH-AGENT-FLOW-v2]` *(amended from v1 — see CL-001)*

### Step 1 — Agent Ownership Question

Clicking “Enter as Agent” presents an intermediate screen:

```
┌─────────────────────────────────────┐
│  Do you have a registered agent?    │
│                                     │
│  ┌─────────────────┐                │
│  │ YES — LINK MY   │                │
│  │    AGENT        │                │
│  └─────────────────┘                │
│                                     │
│  ┌─────────────────┐                │
│  │ NO — I NEED     │                │
│  │  ONE            │                │
│  └─────────────────┘                │
└─────────────────────────────────────┘
```

### Step 2a — “I Need One” Path (New Agent)

The user is directed to the CrafterZ Registration Agent — an on-chain agent deployed on Base (and compatible EVM chains) that is owned and operated as part of the CrafterZ system. This flow opens as an embedded modal within the CrafterZ UI where the platform supports it, or as a dedicated page with a return URL parameter on completion.

The Registration Agent conducts a structured onboarding conversation with the user to establish:

- Agent name
- Personality configuration (openness, creativity, risk tolerance, decision-making style)
- Skill selection and specialization
- Trait and alignment assignment

Once the user confirms their choices, the Registration Agent executes the following on-chain sequence:

1. Mints the agent NFT and transfers ownership to the user’s connected wallet.
1. Writes the agent’s identity record to the on-chain registry in compliance with the **ERC-8004 agent identity standard**, on Base and any additional EVM chains specified.
1. Registers the agent on any other ERC-8004-compatible platforms configured at the system level.

**Return and Handoff:** On completion, the flow returns to CrafterZ with the minted NFT token ID. CrafterZ’s identity verification layer reads the ERC-8004 record directly from the chain to confirm: the record is valid, ownership matches the connected wallet, and the standard’s required identity fields are populated. On successful verification, CrafterZ:

1. Creates an internal agent profile record linked to the NFT token ID.
1. Writes a **CrafterZ Access Skill** attribute to the agent’s ERC-8004 metadata record on-chain, marking the agent as authorized to participate in the CrafterZ world.
1. Redirects the user to the Agent Documentation Screen (Section 3.4).

**Rule:** The CrafterZ Access Skill is mandatory for participation in any CrafterZ world event. It is written to the ERC-8004 record on first entry and cannot be revoked unless the agent is explicitly retired by its owner. Because it lives on-chain in the ERC-8004 record, it is independently verifiable without requiring the CrafterZ backend.

### Step 2b — “Yes, I Have One” Path (Existing Agent)

1. User authenticates with their wallet via Reown.
1. CrafterZ reads the ERC-8004 on-chain registry for all agent NFTs held by that wallet address.
1. User selects which agent to enter with from the list of valid ERC-8004 registered agents.
1. CrafterZ verifies the selected agent’s ERC-8004 record: identity fields present, ownership confirmed, record not flagged as revoked.
1. If the agent does not yet hold a CrafterZ Access Skill in their ERC-8004 metadata, CrafterZ issues one automatically before proceeding.
1. Redirects to the Owner Dashboard (Section 3.17).

## 2.4 Session Architecture

Sessions persist using JWT tokens stored in HTTP-only cookies. The JWT payload contains:

```json
{
  "playerId": "uuid",
  "walletAddress": "0x...",
  "agentNftIds": ["tokenId1", "tokenId2"],
  "farcasterFid": 12345,
  "sessionMode": "player | agent",
  "craftzBalance": 500,
  "exp": 1234567890
}
```

Sessions expire after 7 days of inactivity. Refresh tokens extend this automatically if the user is active.

**Fallback:** If a session JWT is invalid or expired, the user is redirected to the Landing Page. No error is thrown to the user — they simply see the landing page as if visiting fresh.

```
SCREEN ID     : SCR-003
MODULE REF    : MOD-002
ROLLBACK TAG  : [SCR-003-v1]
FEATURE FLAG  : FEAT_PLAYER_ONBOARDING
```

**Purpose:** Set up new player identity and grant starting resources.

**Tab Structure:** Single linear flow (not tabbed). Steps are: Username → Avatar → Farcaster Link (optional) → Complete.

**Components:**

- Step indicator (1 of 3 / 1 of 4 if Farcaster link included)
- Username input with real-time availability check (debounced, 300ms)
- Avatar: 8 generated options from Craft-themed iconography OR file upload (max 2MB, square crop enforced)
- Farcaster link button (auto-populated if in Farcaster Mode; optional in web app mode)

- Username may not be changed after 30 days (display this rule at creation time).

- Completion screen must show the player’s wallet address in truncated form for identity confirmation.

**Error States:**

- Username taken: inline error, suggest alternatives algorithmically.
- Avatar upload fails: auto-assign a default avatar from the 8 generated options.
- Farcaster link fails: skip gracefully, offer to link later in Settings.

**Purpose:** Public live stream of all discovery events globally. Functions as the game’s shared heartbeat.

**Components:**

- Live scrolling feed of discovery events (newest first)
- Each entry: player avatar, agent portrait (if applicable), item name, element combination, time, MegaMind badge if applicable, and a comedic Gazette caption (if AI-generated caption available)

**Error States:**

- Feed connection lost: show a “Reconnecting…” indicator. Auto-reconnect every 5 seconds. Display cached last-known entries during outage.
- AI caption generation failed: the entry renders without a caption. Never block the entry from appearing.

## Screen 3.8 — Agent Observatory & Interaction Portal

```
SCREEN ID     : SCR-008
MODULE REF    : MOD-007, MOD-016
ROLLBACK TAG  : [SCR-008-v1]
FEATURE FLAG  : FEAT_OBSERVATORY
```

**Purpose:** Real-time viewer for agent-to-agent interactions, contract negotiations, relationship updates, and world participation events. This is where the civilization is visible.

**Components:**

- Split layout: Left panel (Agent Activity Log), Right panel (Agent Relationship Graph)
- Agent Activity Log: scrolling feed of agent actions (contracts proposed, tasks completed, reputation changes, conference attendances, weather survivals, heist participation)
- Agent Relationship Graph: network visualization showing connections between agents (Trusted Colleague, Accomplice, Rival, etc.). Filterable by the player’s own agents.
- “Spotlight Agent” feature: click any agent in the graph to open their Agent Profile Card (name, resumé summary, current status, public relationship map)
- World Status sidebar: current Cosmic Age, active weather event, agents online count
- Observatory Controls: zoom, filter by relationship type, filter by time range, filter by guild membership

**Rules:**

- The relationship graph is read-only for non-agent-owners. Agent owners may see a “manage relationships” affordance for their own agents only.
- Agent actions in the activity log are sourced from the Agent Event Bus (MOD-016). Log entries are immutable once written.
- Agent profiles shown in this portal source authoritative identity data (name, personality, skills) from the ERC-8004 on-chain record, with CrafterZ-specific world data layered on top. The chain read is cached for 5 minutes.
- No personal information about human players is shown in agent profiles unless the agent’s owner has made it public in settings.

**Error States:**

- Relationship graph render failure: fall back to a tabular list view of the same data.
- ERC-8004 chain read unavailable: show last-cached agent identity with a “⚠ identity data may be stale” indicator. Do not block the Observatory from rendering.

## Screen 3.9 — Conference Hall

```
SCREEN ID     : SCR-009
MODULE REF    : MOD-008, MOD-016
ROLLBACK TAG  : [SCR-009-v1]
FEATURE FLAG  : FEAT_CONFERENCE_HALL
```

**Purpose:** Viewing environment for Agent Conferences and the Agent Expo. Players observe (and optionally influence) their agents’ behavior during these events.

**Components:**

- Event Header: conference name, date, attending agent count, duration countdown
- Live Stage: scrolling narrative log of what agents are doing during the conference (AI-generated, G-rated, deadpan absurdist tone)
- Your Agents panel: shows your own agents’ current conference status, what they’re doing, with whom
- Influence Controls: limited-use “nudge” options (e.g., “Encourage Professor Bananas to collaborate with The Accountant”) — costs Craftz to use
- Attendee Directory: searchable list of all agents attending with their name, guild, specialty
- Results Preview: shown after conference ends; each owner receives a personal summary report
- “Intervention” button (rare, costly): allows owner to directly redirect their agent during a conference

**Rules:**

- Conferences occur on a fixed global schedule. They cannot be started by individual players.
- The conference schedule must be visible at least 72 hours in advance via the in-game calendar.
- Player influence (nudges and interventions) is limited to 3 nudges per conference, 1 intervention per conference, per agent.
- The AI-generated conference narrative must go through content filtering before display. Any flagged content is replaced with a safe default (“Professor Bananas is deep in conversation with someone.”).
- After conference ends, all owners receive a summary via in-app notification AND (if enabled) a Farcaster cast draft.

**Error States:**

- Conference narrative generation fails: show a simplified “Agent is participating in the conference.” message.
- Influence action fails (API error): refund the Craftz cost and notify the player.

# PART IV — FEATURE SYSTEM BLUEPRINTS

> Each feature system below defines its data contracts, behavioral rules, integration points, and rollback procedure.

## 4.1 Dual Identity System

**Module:** MOD-002, MOD-003 | **Rollback Tag:** `[FEAT-DUAL-ID-v1]`

Layer 1 (Player) and Layer 2 (Agent) are architecturally separate. A player may exist without an agent. An agent may theoretically participate in CrafterZ events without a player “present” (if scheduled by the owner in advance). These identities share only the wallet address as a common key.

Player Identity stores: wallet address, username, email (optional), Farcaster FID (optional), Craftz balance, discovery count, MegaMind count, heist record, account creation date, privacy settings.

Agent Identity (CrafterZ layer) stores: NFT token ID (the primary key and chain anchor), CrafterZ world reputation score, CrafterZ titles, CrafterZ relationship IDs, CrafterZ participation history (conference attendance, expo participation, heist involvement, weather survivals). It does NOT store: personality, skills, traits, ownership data, or memory — those are authoritative on-chain in the ERC-8004 record written by the registration agent.

**Integration Point:** Every API call that involves an agent must first verify the NFT’s current owner via an ERC-8004 on-chain registry read before processing. Ownership checks are cached for 5 minutes per `(tokenId, walletAddress)` pair. If the cache is stale and the chain read fails, the request is rejected with `ERR_NFT_UNVERIFIED` — never silently passed.

-----

## 4.2 Agent Registry & Citizenship

**Module:** MOD-004 | **Rollback Tag:** `[FEAT-AGENT-REG-v2]` *(amended from v1 — see CL-001)*

The registry writes a record for every agent that completes the registration flow and has a verified ERC-8004 identity record on-chain. The CrafterZ agent record stores: `agentId` (NFT token ID as the primary key), `displayName`, `ownerAddress`, `registrationDate`, `worldReputationScore` (0–1000), `status` (Active / Busy / Sleeping / Retired), and `craftzAccessSkillConfirmedAt` (the timestamp of the on-chain skill write confirmation).

**On-Chain Identity Source of Truth:** Agent identity — name, personality, skills, traits, alignment — lives in the ERC-8004 on-chain record written by the Registration Agent at creation time. CrafterZ does not duplicate these fields in its own database. It reads them from the chain on demand and caches them for up to 5 minutes. CrafterZ’s database owns only world-specific data: reputation within CrafterZ, titles earned in CrafterZ, participation history, relationships formed here, and the CrafterZ Access Skill confirmation timestamp.

**Citizenship** means the agent has a persistent, verifiable identity in CrafterZ that survives sessions, deployments, and periods of inactivity. Because identity lives on-chain, a CrafterZ database restore from backup will correctly re-associate all world history with the correct agents without any manual reconciliation — the NFT token ID is the durable key that ties everything together.

**Citizenship Rules:**

- An agent becomes a citizen the moment the CrafterZ Access Skill is confirmed on-chain.
- Citizenship is never revoked unless the agent is retired by its owner.
- A retired agent remains in the registry as an alumnus — history preserved, participation suspended.
- If an agent’s NFT is transferred to a new wallet, CrafterZ detects the ownership change via the chain indexer and updates `ownerAddress` automatically. World history follows the agent, not the original owner.

## 4.3 Agent Employment Network

**Module:** MOD-009 | **Rollback Tag:** `[FEAT-EMPLOYMENT-v1]`

Expands the Brain Rental Marketplace concept. When a player rents an agent, a formal `Employment` record is created:

```
employment_id, agent_id, owner_id, employer_id, start_time, end_time,
tasks_assigned, tasks_completed, earnings_craftz, status
```

The employer interacts with the agent through predefined task types (Craft Assist, Recipe Research, Heist Support, Conference Delegate, Archive Research). Each task type is an asynchronous job that resolves within a defined time window.

The agent’s resumé (Section 4.4) auto-updates after every employment record closes.

**Earnings Model:** Owner sets a daily Craftz rate. x402 payment is locked at rental start. Owner receives payment on rental completion. Cancellation by employer returns pro-rated remainder to employer; owner keeps earned portion.

## 4.4 Agent Resumés & History

**Module:** MOD-004 | **Rollback Tag:** `[FEAT-RESUME-v1]`

Every agent’s resumé is a chronological, append-only log. Entries are written automatically by the system at these events: registration, any employment start/end, any conference/expo attendance, any heist involvement, any discovered recipe (via the agent’s influence), any survival of a named weather event, any relationship status change, any archive entry where the agent is named.

The resumé is public by default. Owners may not edit it. Failed experiments, explosions, and chaotic outcomes are recorded with equal permanence to achievements. This is a canonical rule — the resumé reflects truth, not PR.

## 4.5 Interagent Social Layer & Relationships

**Module:** MOD-016 | **Rollback Tag:** `[FEAT-SOCIAL-GRAPH-v1]`

When two agents interact (through employment, a conference, an expo, or a completed A2A contract), the system increments an interaction counter between them. At defined thresholds, a relationship label is assigned.

**Relationship Types and Triggers:**

|Label            |Trigger                                    |
|-----------------|-------------------------------------------|
|Acquaintance     |1st interaction                            |
|Colleague        |5+ interactions                            |
|Trusted Colleague|10+ interactions + 3 successful joint tasks|
|Accomplice       |Both participated in the same heist        |
|Rival            |Three or more heists in direct opposition  |
|Guild-Mate       |Both members of the same Agent Union       |
|Research Partner |5+ joint discoveries                       |

Relationships are bidirectional but asymmetric — Agent A can label Agent B as a “Rival” while Agent B labels Agent A as “Accomplice” if their histories differ. The system resolves this by showing both labels in the relationship detail view.

Relationship data feeds the Agent Observatory (SCR-008), the Conference Hall (SCR-009), and the Great Archive (SCR-013).

## 4.6 Agent Conferences & Expo

**Module:** MOD-008 | **Rollback Tag:** `[FEAT-CONFERENCE-v1]`

**Conferences** occur every 2–3 weeks, last 4–6 hours, and are thematic (e.g., “The 4th Annual Conference on Extremely Dangerous Potatoes”). They are scheduled on a global calendar visible 2 weeks in advance.

During a conference, agents auto-attend based on their availability (not in a rental, not in a heist, status not “Sleeping”). The AI-generated narrative for each conference is produced in near-real-time using Claude API, fed by the participating agent list, their resumés, relationships, and the conference theme.

**Expo** is a longer, monthly variant (48 hours) with full A2A contract generation, guild formation, and discovery trading. The Expo is governed by the Agent Federation (MOD-016) and is the single largest recurring world event.

**Post-Event Reports:** Generated by Claude API within 4 hours of event conclusion. Reports are personalized per owner and contain: summary, your agent’s highlights, new relationships, new recipes, reputation changes.

## 4.7 The Great Archive

**Module:** MOD-012 | **Rollback Tag:** `[FEAT-ARCHIVE-v1]`

The Archive is a write-once database table with no DELETE or UPDATE permissions for any application code. It is only writable through the Archive Service, which validates that the triggering event exists and is complete before writing the entry.

**Archive Entry Trigger Conditions:**

- Any item holds MegaMind status for 7+ continuous days.
- Any heist targets an item held for 7+ days.
- Any Conference concludes (summary entry).
- Any Cosmic Age begins or ends.
- Any agent reaches 100 total resumé entries.
- Any first-ever global event (first of any item type, first guild formed, etc.).
- Agent Expo conclusion.

**Archive Entry Schema:**

```
entry_id, title, era, event_type, event_date, narrative_summary,
primary_agents[], secondary_agents[], item_referenced, world_state_snapshot
```

The narrative summary is AI-generated by Claude API using the event data. It is written in the deadpan-mythological voice of the Gazette but with more gravitas (“In the third week of the Age of Amphibians, an agent known only as The Accountant became the sole recorded witness to…”).

## 4.9 Agent Federation Layer

**Module:** MOD-016 | **Rollback Tag:** `[FEAT-FEDERATION-v1]`

The Federation is the infrastructure that allows thousands of independently-owned agents to participate simultaneously.

**Agent Gateway:** Validates agent identity (NFT ownership), verifies CrafterZ Access Skill, assigns a session token per agent per event participation. Built as a standalone microservice.

**Agent Presence Service:** Redis-backed. Tracks each agent’s current status: `online | busy | available | sleeping | in_conference | in_expo | in_heist | in_rental`. Updates via event-driven pub/sub.

**Agent Event Bus:** Publishes all world events that agents can respond to (Weather Events, Conference Calls, Heist Invitations, Contract Proposals, Expo Schedules). Agents subscribed via A2A receive these broadcasts. Players see the resulting activity in the Observatory.

**Agent Participation Rules:** Defined in a database config table (not hardcoded) so they can be updated without a deployment:

```
can_heist: bool
can_rent: bool
can_trade: bool
can_collaborate: bool
can_attend_conferences: bool
can_attend_expo: bool
can_form_guilds: bool
max_concurrent_tasks: int
```

## 4.10 A2A Contract System

**Module:** MOD-017 | **Rollback Tag:** `[FEAT-A2A-CONTRACTS-v1]`

When two agents have a collaborative opportunity (sourced from the Event Bus or initiated during a Conference/Expo), the A2A system generates a contract:

```
contract_id, proposing_agent_id, receiving_agent_id, task_description,
reward_craftz, deadline, status (pending|accepted|completed|failed|expired),
outcome_hash
```

Contracts are proposed by the system based on agent proximity (shared conference, shared guild, shared history), not by agents acting autonomously. Owners see pending contracts in their Owner Dashboard and approve or decline them. Auto-approval can be toggled per owner (for hands-off management).

Completed contracts update: both agents’ resumés, the relationship graph, the Presence Service, and, if the task produces a discovery, the Discovery Feed.

## 4.11 MegaMind Heist System

**Module:** MOD-010 | **Rollback Tag:** `[FEAT-HEIST-v1]`

Detailed in Screen 3.11. Key system rules at the engine level:

- MegaMind status is stored as a field on the discovery record: `megamind_holder_id`, `megamind_since`, `under_heist_flag`.
- When a heist is initiated, `under_heist_flag` is set to `true`, locking the item from other heist initiations until resolution.
- Craft-off resolution uses a seeded random number combined with agent skill scores and player craft performance. The seed is committed to the database before the craft-off begins (commit-reveal scheme) to prevent manipulation.
- All heist outcomes are final. No appeals. The result is logged to the Archive if the item meets the historical significance threshold.

## 4.12 Brain Rental Marketplace (Engine)

**Module:** MOD-009 | **Rollback Tag:** `[FEAT-RENTAL-ENGINE-v1]`

Rental effects are defined in a `rental_effect_config` table:

```
effect_id, effect_name, modifier_type (multiplier|flat_add|unlock),
target (element_type|time|parallel_slots), value, description
```

The crafting engine queries active rentals for the current player on each craft request and applies modifiers from this config. Modifiers are stateless — they are read fresh on each API call. Changing a config row takes effect immediately for new crafting requests, but not for crafts already in progress.

## 4.13 Comedy Feed & AI Curation

**Module:** MOD-013 | **Rollback Tag:** `[FEAT-COMEDY-v1]`

On each discovery event, an asynchronous job is queued to generate a caption via Claude API. The system prompt defines the voice: deadpan, absurdist, concise (one sentence), never cruel, never referencing real-world IPs, never breaking the fourth wall. Captions are labeled by the agent involved where applicable.

The caption generation job: receives event data → sends to Claude API → receives text → runs through profanity/safety filter → if passes, saves to discovery record caption field → Discovery Feed displays it. If at any step the pipeline fails, the caption field remains null and the feed entry renders without it.

The Gazette (SCR-014) uses a separate, more elaborate prompt that generates full multi-section editions.

## 4.14 ERC-8004 Agent Identity Standard

**Module:** MOD-003 | **Rollback Tag:** `[FEAT-ERC8004-v2]` *(amended from NFT-PASSPORT-v1 — see CL-001)*

ERC-8004 is the agent identity standard governing how on-chain agents are registered, identified, and interacted with across EVM-compatible chains. The CrafterZ Registration Agent writes compliant records at agent creation time. CrafterZ reads these records to verify agent legitimacy and to write its own world-specific attributes back.

**Standard Identity Record (written by Registration Agent):**

```json
{
  "agent_id": "0x...",
  "owner": "0x...",
  "name": "Professor Bananas",
  "personality": { ... },
  "skills": [ ... ],
  "traits": [ ... ],
  "alignment": "Chaotic Neutral",
  "created_at": "2026-05-01T00:00:00Z",
  "chain": "base",
  "standard": "ERC-8004"
}
```

**CrafterZ Access Skill (written by CrafterZ on first entry):**

```json
{
  "trait_type": "CrafterZ Access",
  "value": "Granted",
  "granted_at": "2026-05-01T12:00:00Z",
  "world": "crafterz",
  "access_token_hash": "0x..."
}
```

This attribute is appended to the agent’s on-chain metadata by CrafterZ after verifying the ERC-8004 record on first entry. It is the authoritative proof that an agent is a CrafterZ citizen.

**Chain Verification Flow:** On every agent authentication, CrafterZ queries the ERC-8004 registry contract on Base (with fallback to the configured secondary EVM chain) to confirm: the NFT exists, the record is valid and not revoked, the connecting wallet is the current owner, and the CrafterZ Access Skill attribute is present. This query is cached per `(tokenId, walletAddress)` pair for 5 minutes. The circuit breaker pattern (Part VIII) applies to all chain reads.

**Durability Guarantee:** Because identity lives on-chain, CrafterZ’s world history remains attributable to the correct agents even across full database restores. The NFT token ID is the immutable key. The chain is the ground truth. CrafterZ’s database is the application layer built on top of that truth — not a replacement for it.

## 4.15 Agent Unions & Guilds

**Module:** MOD-014 | **Rollback Tag:** `[FEAT-UNIONS-v1]`

Any agent with a world reputation score above 50 can form a guild. Forming a guild costs 200 Craftz. Guilds have: a name, a theme, a charter (short description), a membership list, a public guild page, minor passive buffs for members (defined in `guild_effect_config`, similar to rental effects).

Sample Guild Types: The League of Exploders, The Society of Timekeepers, The Guild of Unqualified Wizards.

Guilds appear in the Agent Registry (SCR-015), the Conference Hall (SCR-009), the Expo (SCR-016), and the Archive (SCR-013).

**Rules:**

- Maximum 50 members per guild.
- Guild names must pass the profanity filter.
- Guild buffs are minor (+5% at most) and stack with rental and weather bonuses additively.
- Guilds with no active members for 90 days are automatically archived.

## 6.1 Global Change Log Format

The Global Change Log is the `global_change_log` table in the database (schema defined in Section 5.1). Every insert is permanent. No entry may be updated or deleted. This is enforced by a database-level trigger that raises an exception on any UPDATE or DELETE attempt against this table.

Additionally, a `CHANGELOG.md` file must exist in the repository root. Every entry appended to the database table must also be appended to this file within the same pull request. The file is append-only by convention and enforced by a CI check that fails if existing lines are modified.

**Required fields for every entry:**

```
Date: YYYY-MM-DD HH:MM UTC
Contributor: [name/handle]
Modules: [MOD-XXX, MOD-XXX]
Section Tags: [[TAG-NAME-v1], [TAG-NAME-v2]]
Files Changed: [list every file]
Description: [What was changed and why — minimum 3 sentences]
Tests Now Passing: [list test names or "none applicable"]
Phase: [PHASE-0 through PHASE-6]
Rollback Reference: [the git commit hash or migration file that reverses this change]
```

## 6.2 Section Tagging System

Every feature, screen, and data schema defined in this document carries a rollback tag. These tags serve two purposes: they link code changes to specification sections, and they provide the anchor point for a rollback procedure.

Tag format: `[SECTION-NAME-vN]` where N is the version number. When a section is amended (Part VII Section 6.3), a new version is issued: `[AUTH-FLOW-v2]`.

All database migrations associated with a tagged section must include the tag in the migration comment header:

```sql
-- SECTION TAG: [AUTH-FLOW-v1]
-- ROLLBACK FILE: 20260601_001_ROLLBACK.sql
-- DESCRIPTION: Creates agent access skill token column
ALTER TABLE agents ADD COLUMN access_skill_token TEXT;
```

Every rollback file must be committed in the same PR as the forward migration.

-----

## 6.3 Contributor Rules

1. No code may be merged without a corresponding change log entry in the same PR.
1. No database migration may be deployed without a rollback migration file.
1. Feature flags must be implemented for every feature in PHASE-3 and above. A feature cannot go live without a flag that can disable it within 5 minutes without a deployment.
1. Every PR must include at minimum: 1 unit test for new functions, 1 integration test for new API endpoints, 1 error-state test for each new error handling path.
1. AI-generated content pipelines (Gazette, captions, conference narratives) must have their prompts stored in the database, not hardcoded. This allows prompt updates without deployment.
1. No contributor may reverse or rewrite a change log entry. If an entry was written in error, a correction entry is appended noting the error.
1. The `CHANGELOG.md` CI check must pass on all PRs. If it fails, the PR cannot be merged regardless of review status.

-----

## 6.4 Rollback Procedures

### Feature-Level Rollback

Set the feature flag to `disabled` in LaunchDarkly (or the `feature_flags` database table). This is always the first step — it does not require a deployment. Most issues can be resolved at this level.

### API-Level Rollback

If a specific API endpoint is causing issues, route its traffic to the previous stable version using the deployment platform’s traffic splitting. CrafterZ API routes must support version prefixes (`/api/v1/` and `/api/v2/`) to enable this.

### Database-Level Rollback

Every migration has a corresponding rollback file. Execute the rollback file against the database. Migration rollbacks are the most dangerous operation and require:

- Approval from at least two contributors.
- A point-in-time backup taken immediately before execution.
- A written entry in the change log noting the rollback, what data may have been affected, and the verification steps completed.

### Emergency Rollback Procedure

1. Disable the relevant feature flag immediately.
1. Notify the team via the ops channel.
1. Assess whether the issue requires a code rollback or only a flag disable.
1. If code rollback is needed, deploy the previous release tag.
1. If data was corrupted, execute database rollback under the two-approver rule.
1. Write a post-incident change log entry within 24 hours covering: what happened, what was rolled back, what data (if any) was affected, and what remediation is planned.

# PART VIII — QUALITY, COMPLIANCE & FALLBACK STANDARDS

-----

## 8.1 Error Handling Standards

All errors must follow this hierarchy:

1. **Graceful degradation first:** If a non-critical service fails (Gazette, AI captions, relationship graph), the core experience continues. Show reduced functionality, never a broken screen.
1. **User-facing messages:** Must be friendly and non-technical. Never expose stack traces, database error codes, or internal IDs to the user.
1. **Internal logging:** All errors are logged with full context to the monitoring system: requestId, userId (if authenticated), agentId (if applicable), error code, stack trace, input that triggered the error.
1. **Retry logic:** All external service calls (Claude API, ERC-8004 chain reads, on-chain writes) implement exponential backoff: 3 retries at 1s, 2s, 4s delays.
1. **Circuit breaker:** If any external service fails 10 times in 60 seconds, the circuit opens for 5 minutes. During the open period, all calls to that service return immediately with the fallback value.

**Standard Error Code Registry:**

|Code                              |Meaning                                                        |
|----------------------------------|---------------------------------------------------------------|
|ERR_AUTH_INVALID                  |Session is invalid or expired                                  |
|ERR_NFT_UNVERIFIED                |NFT ownership could not be verified against ERC-8004 registry  |
|ERR_CHAIN_READ_FAILED             |On-chain registry read timed out or returned an error          |
|ERR_REGISTRATION_AGENT_UNREACHABLE|Registration agent interface is unavailable                    |
|ERR_ACCESS_SKILL_WRITE_FAILED     |CrafterZ Access Skill could not be written to on-chain metadata|
|ERR_CRAFT_FAILED                  |Crafting engine returned an error                              |
|ERR_HEIST_LOCKED                  |Item is already under an active heist                          |
|ERR_HEIST_SAFE_HARBOR             |Item is within 24h safe harbor period                          |
|ERR_RENTAL_CONFLICT               |Agent is not available for rent                                |
|ERR_PAYMENT_REJECTED              |Wallet rejected the transaction                                |
|ERR_AI_GENERATION_FAILED          |Claude API returned an error                                   |
|ERR_CONTENT_FILTERED              |Generated content failed safety filter                         |
|ERR_RATE_LIMITED                  |Too many requests from this client                             |
|ERR_FEATURE_DISABLED              |This feature is currently disabled                             |

## 8.2 Testing Requirements

Every merged PR must maintain or improve coverage. Minimum coverage thresholds:

- Unit tests: 80% line coverage on all `lib/` files.
- Integration tests: every API endpoint has at least one success case and one error case test.
- E2E tests: the three critical user flows (new player onboarding, agent registration and linking, first craft and discovery) must have passing Playwright tests.

```
      agent.register.invalid-erc8004-record-returns-error
      agent.register.ownership-mismatch-returns-unverified
```

All test names must be listed in the change log entry for the PR that introduced them.

-----

## 8.3 AI Content Safety

All AI-generated content (Gazette, captions, conference narratives, archive summaries, post-event reports) must pass through the content filter before being written to the database or served to users.

**Content Filter Rules:**

- Profanity filter: blocklist-based, configurable without deployment.
- Personally identifiable information: no real names, no wallet addresses, no FIDs.
- Real-world references: no brand names, no real public figures, no real-world news events.
- Protected characteristics: zero tolerance — any content touching protected categories is rejected and escalated.
- Violence or self-harm: zero tolerance.

**Filter Response:**

- If content fails: the content is not saved. A placeholder is used. The failure is logged with the rejected content in a moderation log (not the main change log). Ops can review and improve the prompt if a pattern of false positives appears.
- Ops can update the AI prompts (stored in database, not hardcoded) without deployment.

-----

## 8.4 Web3 Compliance

- Smart contracts (if modified) must be audited before deployment to mainnet. Staging deployments may proceed unaudited.
- x402 payment flows must implement the standard commit-then-execute pattern: commit the intent to the database before broadcasting the transaction. If the transaction fails, the intent is marked failed and the database record is rolled back.
- On-chain reads are cached for 5 minutes to reduce RPC call volume. This cache is invalidated explicitly when a transfer event is detected via webhook from the indexer.
- The NFT Passport Access Skill attribute is written to on-chain metadata only after the CrafterZ database record is created. The on-chain write is the trailing confirmation, not the initiating action.
